# 尽量一次成功：自定义 Codex 皮肤工作流

这份文档写给两类读者：第一次制作皮肤的人，以及接到“学习这个仓库，打造一个某某主题皮肤”任务的 AI Agent。

“一次成功”不是保证生成图片一次就完全符合审美，而是尽量让需求理解、环境检查、安装、验收和恢复一次走通，把返工限制在视觉微调，而不是反复修流程。

## 0. 先判断用户究竟要什么

“皮肤”“主题”“效果图”“桌宠”经常被混用。执行前先分类：

| 用户真正想要的结果 | 定义 | 本仓库的处理方式 |
|---|---|---|
| 视觉皮肤 | 用背景图和主题色改变 Codex 外观，保留原生界面与交互 | **本仓库的主路径** |
| 效果图 | 一张展示设计方向的界面合成图，不能直接安装 | 只作参考，必须另外生成不带 UI 的背景图 |
| 悬浮组件 / 桌宠 | 独立角色、头像、动画或可交互浮层 | 不属于皮肤；用户未明确要求时绝不添加 |
| 可发行预设 | 其他用户下载后可直接切换的主题包 | 需要额外处理授权、清单、预览与发行检查 |
| 工具流程改造 | 修安装器、托盘、恢复或兼容性 | 先写测试，再改脚本；不能用一张效果截图代替功能验证 |

用户只说“打造一个某某皮肤”时，默认解释为第一类：**一张纯背景图配合 Codex 原生控件**，不是桌宠、悬浮人物、假界面或自动发布的主题包。

## 1. 建立最小成功契约

动手前至少明确下面七项。用户没有指定时采用右侧默认值，不必把所有问题重新抛给用户。

| 项目 | 要明确的内容 | 推荐默认值 |
|---|---|---|
| 平台 | Windows 还是 macOS；Codex 从哪里安装 | 检测当前 OS，使用官方 Codex 桌面应用 |
| 主题内容 | 人物、风景、品牌、抽象风格或用户图片 | 使用用户提供或用户有权使用的素材 |
| 页面范围 | 只改首页，还是任务页也保留背景 | 首页突出，任务页降低干扰 |
| 构图模式 | `auto`、`ambient`、`banner` 或 `off` | `auto`；只有用户明确要横幅时才固定 `banner` |
| 安全区 | 主体放哪里，哪些位置必须低信息 | 根据原生标题、项目选择器和输入框留白 |
| UI 边界 | 是否添加按钮、卡片、宠物、粒子或新交互 | 保留原生 UI，不添加任何独立浮层或假控件 |
| 交付边界 | 只在本机使用，还是提交预设 / Release | 默认本机安装；仓库只放脱敏效果截图 |

只有三类信息缺失时值得暂停询问：目标平台无法检测、用户给出的参考与要求在横幅/沉浸式上互相冲突且 `auto` 无法消解、素材来源或公开发布权利不清楚。其余细节先用安全默认值推进。

## 2. Agent 应按这个顺序执行

1. **读入口而不是遍历全仓库**：依次读根目录 `README.md`、`docs/INDEX.md`、本文件、目标平台的 `SKILL.md` 与验收清单。
2. **输出一份短实施摘要**：一句话确认这是视觉皮肤而非其他产品，再列平台、构图模式、安全区、UI 边界和交付范围。
3. **做环境预检**：确认 OS、Codex 安装来源与版本、Node 版本、仓库已解压、Codex 是否正在运行，以及恢复入口是否可用。
4. **准备纯背景图**：图片不得包含侧栏、标题、卡片、按钮、输入框、Logo、提示词、水印或其他假 UI。
5. **走平台正式入口**：Windows 使用引导 `.cmd`；macOS 使用 `.command`。不要绕过现有安装、日志与恢复机制另写一次性脚本。
6. **导入并应用主题**：确认看见的是图片本身，而不只是主题色变化；确认启动的是皮肤 Launch 入口。
7. **完成验收矩阵**：首页、任务页、输入框、侧栏、窗口缩放、重新启动和 Restore 全部检查后才宣布完成。
8. **最后处理发布**：未经明确要求，不提交背景原图、角色主题包、用户配置、本机 runtime 或包含个人信息的截图。

如果任何一步失败，先读取持久日志和现有状态，再定位根因。不要连续尝试不同脚本、手工改配置或替换官方程序文件。

## 3. 背景图的通用制作规则

- 推荐 `2560 × 1440`、接近 16:9 的 PNG 或高质量 JPEG。
- 背景图只负责视觉，不承担界面功能。
- 主体与原生文字之间需要明确的低信息安全区；不要机械规定所有主题都必须“人物在右、文字在左”。
- 重要面部、Logo 或物体不要紧贴边缘，给不同窗口比例留下裁切余量。
- 高对比小物件不要出现在输入框、标题或按钮附近，否则会看起来像界面脏点。
- 星光、花瓣、粒子等装饰优先画入图片并限制在安全区；确需动态装饰时必须局部作用、`pointer-events: none`，不能跨页面或盖住原生控件。
- 不要用效果截图、带窗口边框的概念图或 README 预览图作为背景输入。
- 生图完成后先单独检查多余文字、畸形细节、错误 Logo、人物特征和不想要的装饰，再进入安装流程。

## 4. 一次成功的验收矩阵

| 维度 | 必须通过的检查 |
|---|---|
| 图片 | 真实背景已经显示，不是只有强调色改变；主体没有意外裁切或瑕疵 |
| 原生 UI | 导航、项目选择器、输入框、发送键和其他原生按钮仍可点击 |
| 可读性 | 标题、侧栏、输入框与任务内容在浅色和暗色环境下有足够对比度 |
| 页面隔离 | 首页可以更有氛围，普通任务页不出现跨区域装饰或大面积视觉干扰 |
| 响应式 | 最大化、较窄窗口和常见缩放比例下没有横向溢出或关键内容遮挡 |
| 生命周期 | 切换页面、刷新或重新启动后皮肤仍能重新应用，不重复叠加元素 |
| 恢复 | Restore 能移除注入、关闭调试会话并重新打开官方 Codex |
| 隐私与仓库 | 效果截图已脱敏；没有提交账号、任务名、聊天内容、runtime 或官方程序文件 |

视觉检查不能只看新建任务页的一张截图。首页好看但任务页难读、输入框出现脏点、窗口缩小时人物被切掉，都不算完成。

## 5. 最常见的通用失败模式

| 失败模式 | 为什么会发生 | 正确做法 |
|---|---|---|
| 把“皮肤”理解成桌宠或浮动人物 | 名词含义不清，Agent 自行扩展产品范围 | 先按第 0 节分类；未明确要求时只做视觉皮肤 |
| 把效果图直接当背景 | 效果图本身已经画入了 Codex UI | 重新生成 UI-free 背景，效果图只用于比较方向 |
| 只改主题色，没有显示图片 | 启动了普通 Codex，或注入没有重新应用 | 使用皮肤 Launch 入口，并从托盘重新应用 |
| 为了“更像设计稿”增加假按钮 | 把视觉合成误当成真实产品实现 | 所有可见控件优先使用 Codex 原生元素，不复制一套假的 |
| 装饰跨过窗口或出现在输入框 | CSS 选择器过宽，或装饰层使用全局定位 | 装饰留在图片安全区；动态层必须限定壳层和页面 |
| 只按一张截图的尺寸调布局 | 固定像素在其他窗口尺寸下会失效 | 使用比例、安全区和多尺寸验收，不以单一分辨率为真相 |
| 安装失败后反复双击不同脚本 | 终端关闭太快，看不到同一个根因 | 使用引导入口和持久日志；一次只验证一个假设 |
| 直接重写用户配置 | 为解决局部冲突破坏了其他项目或外观设置 | 严格 UTF-8、原子写入、最小字段修改、可恢复备份 |
| 把本机启动副本提交到 GitHub | 误把兼容性 runtime 当成项目依赖或主题资产 | `runtime/` 永久忽略，并在提交和 Release 前检查远端树 |
| 截图泄露个人信息 | 只关注视觉，没有检查侧栏、账号和聊天内容 | 发布前遮挡或换用干净演示环境 |
| 角色、人物或品牌素材直接发行 | 代码许可被误认为图片授权 | 软件许可与素材权利分开；无明确权利时只做本地示例 |
| Codex 更新后旧选择器失效 | 桌面应用 DOM 与启动方式会变化 | 重新运行验证；把适配修复和视觉微调分开处理 |

## 6. 可直接复用的通用提示词

把方括号中的内容替换成你的主题即可：

```text
学习这个仓库及其文档：[仓库链接]

在当前电脑上从零制作一套“[主题名称]”Codex 视觉皮肤。

除非我的要求与仓库默认契约冲突，否则：
- 这里的“皮肤”指背景图、主题色和原生界面的协调，不是桌宠、悬浮组件或效果图。
- 先检测平台和环境，再用对应平台的正式安装入口，不修改官方程序文件。
- 使用一张不带侧栏、按钮、输入框、文字、Logo 或其他假 UI 的纯背景图。
- 保留 Codex 原生导航、项目选择器、任务内容、输入框和所有可点击控件。
- 根据主题主体自动规划安全区；首页突出氛围，普通任务页优先保证文字可读。
- 不新增独立按钮、卡片、宠物、头像、粒子或跨页面装饰，除非我明确要求。
- 安装失败时先读取持久日志并定位根因，不要反复猜测修改。
- 完成后验证首页、普通任务页、输入框、侧栏、窗口缩放、重新启动和 Restore。
- 默认只在本机安装。未经我明确要求，不提交背景原图、成品主题包、本机 runtime 或含个人信息的截图。

开始执行前先给出一份简短实施摘要；只有平台、构图模式或素材权利存在关键歧义时才询问我。
```

用户也可以只写“学习这个仓库，打造一个某某皮肤”。仓库内的 Agent 说明会采用上述默认契约；补充平台、横幅/沉浸偏好和素材来源，成功率会更高。

## 7. 哪些部分仍然不可能保证一次命中

- 审美偏好：人物大小、色温、留白和细节取舍需要人的判断。
- 生成图片随机性：多余文字、错误特征和局部瑕疵仍可能需要重生成。
- 外部版本变化：Codex、Windows Store、macOS 或 Node 更新可能改变启动与 DOM 行为。
- 素材权利：工具可以检查流程，不能替用户获得第三方角色、人物或品牌授权。

因此合理目标是：**工程流程一次跑通，视觉最多一次定向微调，任何失败都能被看见、解释和恢复。**
